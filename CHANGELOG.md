# Changelog

## 2.0.0 — 2026-07-12

大改版：從「2 個確認點的全自動流程」重構為「7 步引導式流程」。用戶回饋：每一步都要跟用戶解釋現在要幹嘛、給附推薦預設的選項。

- **七步架構**：①定題 ②研究（宣告後進行）③大綱確認 ✅ ④案例採訪 ⑤撰文＋humanize（宣告後進行）⑥品檢定稿 ✅ ⑦發佈回報。1/3/4/6/7 停下來給選項（推薦選項排第一），2/5 為「宣告式」——講清楚預設值、邀請覆寫、不等待直接進行。
- **通用化**：主題建議不再預設用戶有課程或任何特定內容——從 AI 對該用戶的既有認識（記憶、先前對話、站上內容）挖建議，一無所知就開放詢問；第一手素材掃描的措辭同步通用化。
- **放寬豐富度規定**：「每個 H2 必須有一個具體物件」改為「在有價值的地方放」——how-to 和比較段落大方放，純概念或過場段落允許只是說清楚；品質檢查項同步改寫。
- README 重寫定位（移除「只有 2 個確認點」賣點，改為七步引導＋推薦預設）；批次模式改為「停下來的步驟整批處理」。

## 1.2.0 — 2026-07-12

用戶回饋 1.1.2 的表格式盤問太囉嗦，改為「採訪式案例蒐集」（新增 Phase 1.5 Case Interview）：

- **挖掘探索式提問**：大綱通過後，一句開放式問題問所有缺案例的段落（「你有沒有相關的經驗？想到什麼講什麼」）；用戶講了就像採訪一樣追問挖深（每案最多 1–3 個短問題），夠寫一段就停。
- **不想細講有台階**：提供「給大方向讓 AI 編」「AI 提方向再編」「改通用情境」「跳過」等選項。
- **落筆前逐字確認**：AI 編的或超出用戶原話的案例，寫進文章前必須原文播回確認「這樣寫 OK 嗎？」；用戶同意具體文字後可用第一人稱——同意即認領。
- 確認點 1 的 Real-example requests 表格簡化為一行狀態（✓ 有素材／? 大綱通過後採訪），訪談不算確認點、是對話。

## 1.1.2 — 2026-07-12

用戶再次修正 1.1.1 的力道不足：「讓它確實是來問我；真的要編也必須用戶同意再編。」

- **Real-example requests 從「需要時附上」改為硬性必附**：大綱含任何第一人稱內容時，確認點 1 必須逐欄列出素材覆蓋狀況，MISSING 的欄位由用戶三選一：(a) 提供真實細節 (b) 明確同意改用通用情境（絕不用第一人稱）(c) 刪掉該欄位。
- **未經用戶明確同意，禁止編任何案例**——含「善意的」通用示意也要先取得同意。
- **硬性關卡**：任何第一人稱欄位未解決（有來源／用戶提供／用戶同意通用／刪除）前，不得進入 Phase 2 撰文。

## 1.1.1 — 2026-07-12

第二篇實戰（〈Claude Skills 是什麼〉）後用戶抓到的漏洞：skill 要求第一人稱案例，卻沒禁止「代編細節」，導致 AI 把素材裡沒有的數字、檔案內容、學員軼事編出來掛在作者名下。

- **禁止虛構真實經驗**（Phase 2 新增 Real experiences only）：第一人稱故事只能來自第一手素材或用戶親口提供；素材證明「故事存在」不等於可以編「故事細節」；缺的細節在確認點 1 一次問齊；用戶沒有案例就寫成明示的通用情境，不得用第一人稱。
- **確認點 1 新增 Real-example requests 區塊**：列出每個第一人稱欄位的素材覆蓋狀況（✓ 有來源／MISSING 請提供或回「none」），維持 2 個確認點不變。
- **品質檢查第 10 項加嚴**：第一人稱描述中的每個具體細節（數字、價格、檔案內容、軼事）都要能追溯到來源，無來源即 fail。

## 1.1.0 — 2026-07-12

第一次實戰跑完整流程（〈使用 AI 的五個層級〉）後的修正。核心教訓：預設寫法太學術、內容只有框架沒有血肉，另補幾個實務洞：

- **引註白話化**（Phase 2 新增 Citation Style）：數據保留來源＋年份，但期刊名、paper 編號、「et al.」不進正文——改用「誰做了什麼研究」的講法。
- **豐富度硬性要求**（Phase 2 新增 Richness Requirements）：每個 H2 至少一個具體物件——可複製的指令、❌/✅ 對照、生活場景清單、或第一人稱故事；純抽象論述過不了確認點 2。
- **第一手素材掃描**（Phase 1 新增 Step 0）：research agent 出發前先找用戶自己的素材（課程字幕、舊文、逐字稿），用戶自己的框架與原話是最強的 E-E-A-T，外部研究是佐證不是主體。
- **Meta Description 中文長度**：150–155 字元是英文規格，中文以 70–80 全形字為準（Google 對 CJK 約 80 字截斷）。
- **品質檢查修正計數並擴充**：原清單實際列了 16 項卻標成 15；補上「每個 H2 有具體物件」與「簡體字／異體字機械掃描」（單一字元如「换」用肉眼會漏，必須跑 regex），現為 18 項、<15 分自動修。
- **.md 發佈的 Schema 網址**（Phase 3 Option B）：存檔時沒有網域，一律用 `https://example.com/[slug]` 佔位＋frontmatter 註記，並在結案報告提醒替換。
- **zenbu MCP 發佈路徑**（Phase 0 / Phase 3 Option A）：明列 zenbu 站台 MCP（`zenbu_create_article` 等）為已知發佈路徑，步驟與 WordPress 一對一對應。

## 1.0.0 — 2026-07-04

從 [lukehsuhao/luke-claude-skills](https://github.com/lukehsuhao/luke-claude-skills) 抽出為獨立 repo，並做以下補強：

- **humanizer-tw 後備方案**：未安裝 `humanizer-tw` 時不再是未定義行為，直接套用 skill 內建的濃縮規則（Phase 2.5）。
- **MCP 工具名彈性**：`mcp__wordpress__*` 明確標註為範例名稱，執行時以 session 實際可用的 WordPress MCP 工具為準；沒接 MCP 就只提供存 .md 選項（Phase 0 / Phase 3）。
- **站內連結**：MCP 發佈流程加一步——先搜站上既有文章，自然置入 2–3 個站內連結再發文（Phase 3 Option A）。
- **批次模式**：新增 Batch Mode 章節。原本 description 就宣稱支援批次，但流程只涵蓋單篇；現在多主題一次跑，維持總共 2 個確認點（大綱一批確認、成文一批確認）。
